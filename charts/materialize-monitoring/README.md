# materialize-monitoring

Unified observability stack for Materialize: an Alloy-based pipeline,
optional bundled backends (Loki / Thanos / Grafana / Alertmanager), and
the dashboards / alerts / rules that consume the Materialize metrics
surface.

## TL;DR

```bash
helm repo add materialize https://materializeinc.github.io/materialize-monitoring
helm install mz-mon materialize/materialize-monitoring --set profile=bundled-stack
```

### Profile selection

A profile chooses sensible defaults for which subcharts are enabled, how
the Alloy pipeline is wired, and how telemetry egresses. Profiles never
override an explicitly-set value: anything you set further down in this
file wins.

Supported profiles:
  - `datadog-agent`: customer runs Datadog Agents; Alloy emits via the
    local Agent's OpenMetrics check or DogStatsD. No bundled backends.
  - `prometheus-remote-write`: customer remote-writes to Datadog or any
    Prometheus-compatible endpoint. Gateway applies cardinality reduction
    before egress.
  - `otlp`: customer runs an OTel-native backend (Honeycomb-class). Alloy
    emits OTLP to a customer-supplied collector endpoint.
  - `bundled-stack`: customer has no existing observability stack. This
    chart deploys Loki + Thanos + Grafana + Alertmanager locally.

| Name      | Description                                                               | Value           |
| --------- | ------------------------------------------------------------------------- | --------------- |
| `profile` | Deployment profile that drives subchart enablement and pipeline defaults. | `bundled-stack` |

### Subchart enablement (Helm tags)

Subchart enablement has two layers, in precedence order:

  1. **Per-chart circuit breaker (`<chart>.enabled`).** Each subchart
     dependency carries a `condition: <chart>.enabled` in `Chart.yaml`.
     The corresponding `enabled:` key is left commented-out under each
     subchart block below; uncommenting it force-includes (`true`) or
     force-excludes (`false`) that chart regardless of any tag state.
     Helm evaluates `condition:` first and only falls through to tags
     when the path is absent from values, so this is a hard override.

  2. **Helm tags (this `tags:` block).** When no `<chart>.enabled` is
     set, enablement is decided by tags. Each dependency in `Chart.yaml`
     carries a *group tag* (e.g. `bundled-backends`) and a *per-chart
     tag* (e.g. `loki`); a dependency is included if any of its tags
     evaluates true (OR semantics). To opt one chart out of an
     otherwise-enabled group, set the group tag false and flip the
     per-chart tags individually — or use the circuit breaker.

Group ↔ chart mapping:
  pipeline          → alloy-agent, alloy-gateway
  bundled-backends  → loki, thanos, grafana, alertmanager
  cluster-metrics   → kube-state-metrics, metrics-server
  crds              → prometheus-operator-crds

The defaults below match `profile=bundled-stack`. Other profiles flip
`bundled-backends` off and the pipeline exporters appropriately; see
